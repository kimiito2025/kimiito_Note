# Cisco Secure Access（CSA）初心者ガイド

作成日：2026-05-27

## CSAとは（ざっくり理解）

**Cisco Secure Access（CSA）** = シスコのセキュアなアクセス・ゼロトラストソリューション

社員が会社のネットワークやクラウドアプリケーションに接続する際に、デバイス状態・ユーザー身元・アクセス場所を検証して、安全にアクセスさせる仕組みです。

## 従来方式との違い

### 従来のVPN方式
```
社員PC → VPN接続 → 会社内ネットワーク → アプリケーション
         （境界認証のみ）
```
問題：VPN内に入ってしまえば、その後のアクセス制限が弱い。

### Secure Access（ゼロトラスト）方式
```
社員PC → デバイス検証 → ユーザー認証 → 位置情報確認 → 最小限リソースのみ許可
         （継続的検証）
```
特徴：「信頼できない」前提で、常に検証し続ける。

## CSAの主要機能（3ブロック理解）

### ブロック1：ユーザー・デバイス認証
- ユーザー本人か？ → ID/パスワード + MFA（多要素認証）
- デバイスは健全か？ → OS パッチ、アンチウイルス等の状態確認

### ブロック2：アクセス制御・可視化
- 何にアクセスしようとしているか？ → 権限に応じて許可/拒否
- アクセスの詳細をログ記録 → 後で監査・追跡可能

### ブロック3：アプリケーション保護・脅威検出
- 不正な通信パターン検出 → リアルタイム遮断
- データ流出防止 → 機密情報の持ち出し防止

## なぜCSAが必要か

| 課題 | 対策 |
|------|------|
| リモートワーク増加 | 場所を問わず安全にアクセス可能に |
| 社内外ネットワークの混在 | クラウド・オンプレ問わず統一的に保護 |
| 標的型攻撃 | デバイス・ユーザー・通信を多層検証 |
| 内部脅威 | アクセス権の最小化で被害を局限化 |

## CSAの導入で変わること（ユーザー視点）

### 導入前
- VPN接続1回で、社内ネットワーク全域にアクセス可能
- デバイス管理はユーザー任せ
- アクセスログはほぼ不可視

### 導入後
- リソースごとに認証が必要（段階的信頼）
- デバイスのセキュリティ状態を自動チェック
- 全アクセスが監査可能（監査ログ充実）

## CSAの構成要素（実装側）

1. **アイデンティティプロバイダー** → ユーザー身元を統一管理（AD, Okta等）
2. **ゼロトラストゲートウェイ** → 全通信をチェック
3. **デバイス管理** → PC/スマホのセキュリティ状態確認
4. **アナリティクス・SIEM連携** → ログ監視、異常検知
5. **SaaS/プライベートリソースへのコネクタ** → アクセス経路の統一

## よくある質問

**Q: VPNより複雑では？**  
A: 初期設定は複雑ですが、ユーザー体験は「透過的認証」で実は簡単。リソースに応じて自動判定。

**Q: すべてのリソースに対応？**  
A: クラウド（Salesforce、Microsoft 365等）とオンプレミス（社内アプリ等）の両方対応。ハイブリッド環境が得意。

**Q: コストは？**  
A: ユーザーライセンス + リソース形態で課金。大企業向けに最適化。

## ひとこと

**CSA = VPN 2.0**  
昔のVPNは「社内/外」という二者択一。CSAは「デバイス+ユーザー+リソース」を常時検証して、最小限の権限だけ与える。セキュリティと利便性の両立を実現したソリューションです。
