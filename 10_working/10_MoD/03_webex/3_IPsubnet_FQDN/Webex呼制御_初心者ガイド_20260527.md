# Webex Calling 呼制御（Call Control）ガイド

作成日：2026-05-27  
参考：Webex Calling のポート参照情報

## 呼制御とは

**呼制御 = 電話（SIP）通話の開始・終了・経路制御を管理する仕組み**

Webex Calling では、クラウド上の呼制御機能が「誰と誰が通話可能か」「通話の品質設定」「外線への接続」などを決定します。

## 呼制御の流れ（基本）

```
1. 発信者が電話をかける
        ↓
2. Webex Calling クラウド側で呼制御開始
        ↓ （以下、呼制御による判定）
3. 対象ユーザーの権限確認 → 発信可能か？
4. 発信先の属性確認 → 内線？外線？国番号？
5. 宛先ルーティング決定 → どのゲートウェイを経由？
6. メディアパス確定 → 音声・映像の経路を設定
7. 通話開始 & 品質監視
        ↓
8. 通話終了
        ↓
9. CDR（通話記録）を保存 → 請求・監査用ログ
```

## Webex Calling での呼制御の実体

Webex Calling に関連する主要な呼制御要素：

### 1) シグナリング（呼制御信号）
- **プロトコル** → SIP（Session Initiation Protocol）over TLS
- **送信元** → Webex App / 電話機 / ゲートウェイ等
- **送信先** → Webex Calling Cloud（呼制御サーバー）
- **ポート** → TCP 5062, 8934（主）
- **ドメイン** → *.webex.com, *.broadcloud.eu 等（FQDN指定）
- **特徴** → IPアドレスの固定値がないため、URLベース許可が必須

### 2) メディア（音声・映像）
- **プロトコル** → SRTP（Secure RTP）over UDP
- **送信元** → クライアント/デバイス
- **送信先** → Webex Calling Cloud（メディアサーバー）
- **ポート** → UDP 19560～65535（動的割り当て）
- **IP範囲** → Webex Calling Services の IP サブネット（1）参照
  - 例：23.89.0.0/16, 62.109.192.0/18 等
- **特徴** → IPサブネット+ポート範囲で許可

### 3) ローカルゲートウェイ（オンプレミスPSTN接続）
- **役割** → 社内電話システムとWebex Calling を相互接続
- **呼制御信号** → SIP TLS（クラウド側と同一プロトコル）
- **メディア** → SRTP over UDP
- **ゲートウェイ側ポート** → TCP/UDP 5060-5080（SIP）
- **必須許可** → Webex Calling Services の IP サブネット（1）への接続

## 呼制御の許可設定（FWルール例）

### 必須1：シグナリング（HTTPS）
```
Outbound: デバイス/PC → Webex Calling Cloud
Protocol: TCP
Port: 5062, 8934（SIP TLS）, 443（HTTPS）
宛先：*.webex.com, *.broadcloud.eu などのドメイン（URL指定）
重要：IP指定は不可（Webex側IPレンジは動的変更のため）
```

### 必須2：メディア（音声/映像）
```
Outbound: デバイス/PC → Webex Calling Cloud
Protocol: UDP
Port: 19560～65535（動的割り当て範囲）
宛先IP：Webex Calling Services IP サブネット（1）
         例）23.89.0.0/16, 62.109.192.0/18, 85.119.56.0/23 等
```

### オプション：ローカルゲートウェイ対応
```
Inbound: Webex Calling Cloud → ローカルゲートウェイ
Protocol: TCP/UDP
Port: 5062, 8934（呼制御）, 5060-5080（SIP）, 8500-8699（メディア）
送信元IP：Webex Calling Services IP サブネット（1）
```

## 呼制御設定上の注意点

### ❌ よくある誤り
1. **IP固定で呼制御を許可しようとする** → Webex側のIPは動的。この方法では通話が途切れる可能性あり
2. **地域による IP サブセット フィルタリング** → Webexはグローバルサービス。通話品質が著しく低下
3. **NAT/PAT環境でSIP ALG有効のまま** → 呼制御信号が破損。通話できなくなる可能性
4. **プロキシでメディア検査** → メディア品質が低下。UDP SRTPは検査不対応

### ✅ 推奨方法
1. **シグナリング** → URL/ドメインベース許可
2. **メディア** → IP サブネット（1）+ UDP ポート範囲で許可
3. **NAT環境** → SIP ALG 無効化、適切な NTP タイムアウト設定（30分以上推奨）
4. **プロキシ** → シグナリングのみプロキシ経由、メディアは直通化

## Webex Calling の呼制御トラフィック種別

| トラフィック種 | 方向 | プロトコル | ポート | 許可対象 | 備考 |
|---|---|---|---|---|---|
| **呼制御信号** | Out | TCP(SIP TLS) | 5062, 8934 | ドメイン/FQDN | シグナリング登録・通話開始終了 |
| **音声メディア** | Out | UDP(SRTP) | 19560-65535 | IP サブネット（1） | リアルタイム音声 |
| **映像メディア** | Out | UDP(SRTP) | 19560-65535 | IP サブネット（1） | リアルタイム映像 |
| **設定・管理** | Out | TCP(HTTPS) | 443, 8443 | IP サブネット（2）, （3） | デバイス構成・ファームウェア更新 |
| **時刻同期** | Out | UDP(NTP) | 123 | IP サブネット（1） | デバイスの時刻同期（重要） |

## よくある質問

**Q: 呼制御と音声メディアが別経路？なぜ？**  
A: シグナリングはHTTPS（暗号化・信頼性優先）、メディアはUDP SRTP（リアルタイム性優先）で、要件が異なるため。

**Q: IP サブネット一覧を見て、どれがどの地域？**  
A: Webexドキュメントには対応地域が明示されていません。Cisco営業に確認が必要。

**Q: ローカルゲートウェイがなければ、呼制御設定は不要？**  
A: いいえ。Webex Calling との間の呼制御シグナリング・メディアは常に必須です。

**Q: CDR（通話記録）ログはどこ？**  
A: Webex Control Hub の「通話履歴」で確認。課金・監査に使用。

## Webex呼制御の重要ポイント（まとめ）

| 項目 | ポイント |
|------|---------|
| **シグナリング許可** | URL/ドメインベース必須。IP固定では継続性不可 |
| **メディア許可** | IP サブネット（1）+ UDP ポート範囲で統一 |
| **NTP（時刻同期）** | 呼制御・セキュリティに必須。30分タイムアウト推奨 |
| **NAT対応** | SIP ALG無効化、適切なバインディング期間設定 |
| **ローカルゲートウェイ** | Webex Calling ← → オンプレ電話システム の相互接続に必須 |
